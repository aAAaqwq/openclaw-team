# 技术债务管理

量化、跟踪、管理技术债务的系统化方法。

## 债务分类

### 五大类技术债务

| 类型 | 定义 | 典型例子 | 修复难度 |
|------|------|----------|----------|
| **架构债务** | 系统设计不佳 | 循环依赖/耦合过紧/缺少抽象 | 🟢中 |
| **代码债务** | 代码质量问题 | 重复代码/过长函数/魔法数字 | 🔵低 |
| **测试债务** | 测试覆盖不足 | 缺少单元测试/测试不稳定 | 🟡中 |
| **文档债务** | 文档缺失/过期 | 缺少API文档/架构图过时 | 🟢低 |
| **基础设施债务** | 工具链/部署问题 | 手动部署/无CI/旧依赖 | 🔴高 |

### 债务来源分析
```
业务压力（60%）：赶工期 → 代码质量妥协
技术迭代（25%）：旧架构不再适配新场景
团队变动（10%）：人员更替导致知识断层
其他（5%）：忽视编码规范/缺少流程
```

## 量化评估

### SQALE方法
SQALE（Software Quality Assessment based on Lifecycle Expectations）：
```
债务 = 修复成本（人时）× 规模（代码行/模块数）

分级：
- 阻塞级（Blocker）：立即修复，否则无法开发
- 严重级（Critical）：功能异常或有安全风险
- 主要级（Major）：影响可维护性，需计划修复
- 次要级（Minor）：小范围质量问题
- 信息级（Info）：建议优化
```

### 量化公式
```python
def calculate_tech_debt(issues):
    """
    简化技术债务量化
    Weight: Blocker=10, Critical=5, Major=2, Minor=0.5
    """
    weights = {
        'blocker': 10,
        'critical': 5, 
        'major': 2,
        'minor': 0.5
    }
    total_weight = sum(weights.get(i.severity, 0) for i in issues)
    return total_weight * 0.5  # 假设平均修复时间30min(0.5人时)

# 债务率 = 债务总量 / 代码行数 * 1000
debt_ratio = (total_debt / loc) * 1000
# 目标：债务率 < 5（健康）| 5-15（可控）| > 15（危险）
```

## 还债优先级

### 利息率模型
```
债务"利息" = 每延期一周新产生的问题数

高利息 → 优先修复：
- 安全漏洞（利息：灾难级）
- 功能bug影响用户（利息：用户流失）
- 阻塞新功能开发（利息：开发停滞）

低利息 → 可以延期：
- 命名优化
- 代码格式
- 未使用的import
```

### 风险因子评估
```python
def priority_score(issue):
    risk = issue.severity_weight  # 1-10
    interest = issue.weekly_impact_weight  # 1-10
    blocking = 5 if issue.blocks_new_feature else 0
    return risk * 0.3 + interest * 0.5 + blocking * 0.2
```

### 还债推荐策略
```
策略1：清理小债务（20/80法则）
→ 每天15分钟修复Minor/Major级别债务
→ 长期效果显著

策略2：修复高利息债务
→ 每周评估一次高利息债务
→ 列入Sprint Backlog

策略3：大重构前先还债
→ 重大架构变更前先做清理
→ 降低重构风险

策略4：新代码零容忍
→ 新代码引入的债务必须当天修复
→ 防止债务持续累积
```

## 技术债务板（Backlog管理）

### 板模板
```
┌──────────────┬────────────────┬─────────────────┐
│  待评估       │  计划中         │  进行中         │
│  🔴 bug-001   │  🔴 bug-003     │  🔴 bug-002    │
│  🟡 arch-001   │  🟢 doc-001     │                │
│               │                │                │
├──────────────┼────────────────┼─────────────────┤
│  已验证       │  已完成         │  关闭/豁免      │
│  🟢 code-001  │  🟡 test-002     │  doc-002（放弃）│
│               │                │                │
└──────────────┴────────────────┴─────────────────┘
```

### Sprint容量分配
- **每月首个Sprint**：30%容量还债，70%新功能
- **每月其他Sprint**：10%容量还债，90%新功能
- **危机模式**：债务Sprint（全量还债，无新功能）

### 债务豁免标准
只有满足以下条件才能标记为"豁免"：
1. 该债务不会产生安全风险
2. 修复成本 > 三年内预估利息
3. 该代码将在6个月内被替代
4. 团队书面确认接受风险

## 技术债务健康度评估

| 维度 | 健康 | 注意 | 危险 |
|------|------|------|------|
| 债务率 | <5 | 5-15 | >15 |
| 阻塞级问题 | 0 | 1-2 | ≥3 |
| 新代码债务率 | <3% | 3-10% | >10% |
| 修复周期 | <2周 | 2-8周 | >8周 |
| 修复成功率 | >80% | 60-80% | <60% |
