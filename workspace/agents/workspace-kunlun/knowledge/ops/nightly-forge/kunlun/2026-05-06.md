**最优解不是“修一个参数”，而是完成了一次 P0 级收敛：把 Telegram 群聊从“全员乱醒 + 无法稳定回复”的失控态，拉回到“点名唤醒、显式路由、统一模型栈”的可控态。**

### 为什么这是今日最优解

- 它把系统从“任何消息都可能惊动 17 Bot”改回“只有被点名者响应”，这直接恢复了昆仑路由协议的前提。
- 它为后续验证建立了可执行测试序列：**点名昆仑 → 点名专业 Agent → 非点名静默验证**。

### 战略评价
这是一次典型的**先止血、再校准、后验证**。ROI 最高的不是继续追日志，而是先恢复最小可控面。

## 2）失败模式

### 失败模式A：把“群聊可见”误当成“群聊可控”
**根因**：`requireMention:false` 让所有消息都可唤醒全部 Agent，群聊从路由系统退化成广播系统。  
**表现**：`lane wait exceeded`、`stuck session`、`agent cleanup timed out`、超长 `prep stages`。  
**结论**：军团群必须是**定向唤醒系统**，不能是**全员监听系统**。

### 失败模式B：mention 规则不一致，导致“看见消息 ≠ 能被正确点名”
**根因**：`requireMention` 与 `mentionPatterns` 失配。  
**表现**：消息进入 Gateway，但大量被标记为 `no-mention`。  
**结论**：点名机制必须以 **Bot 用户名模式**为主，不可依赖模糊文本猜测。

### 失败模式C：模型配置漂移会把通信问题伪装成业务问题
**根因**：各 Agent 存在显式 `model` 覆盖，与全局默认不一致。  
**表现**：同样的消息链路，进入不同 Agent 后行为不一致。  
**结论**：群聊体系里，**模型优先级必须标准化**，否则排障无法收敛。

### 失败模式D：治理闭环文档缺位，会让恢复工作只能靠会话记忆硬撑
**根因**：`knowledge/ops/` 一度缺失，`memory/decisions/` 当前为空，任务卡只恢复了骨架。  
**表现**：能做事，但难复盘；能推进，但弱验真。  
**结论**：没有 nightly forge / daily reports / decision artifacts，系统会在第二天失去组织记忆。

### 失败模式E：任务追踪恢复了外壳，但未恢复原始任务语义
**根因**：稷下、轩辕两张任务卡目前是追踪卡，不是完整原始任务单。  
**表现**：有 owner、有验收标准，但 deadline、L1确认、真实任务上下文仍待补。  
**结论**：任务管理最怕“卡存在、事实缺失”。这会让 Level 2、Level 3 追踪失真。

## 3）新技能 / 新洞察

### 新技能1：P0 群聊止血四件套
**技能定义**：
- `requireMention: true`
- `visibleReplies: automatic`
- 全量 `mentionPatterns`
- 统一模型优先级

**适用场景**：Telegram 多 Agent 群聊失控、全员 typing、点名失效、排障发散。  
**价值**：先恢复可控响应边界，再讨论更深层 provider / network / session 问题。

### 新技能2：治理骨架优先于效率优化
**洞察**：今天真正恢复的，不只是聊天链路，还有 `knowledge/ops/` 的最小治理框架。  
**结论**：当系统经历迁移或故障后，第一优先级不是“继续扩功能”，而是**先补 nightly / daily / okr / merit 这些组织骨架**。

### 新技能3：决策日志允许临时记在 daily memory，但不能长期停留在那里
**洞察**：`heartbeat-state.json` 已明确记录：当前无 pending 决策超 24h，且 P0 决议暂记于当日 memory；但 `memory/decisions/` 为空。  
**结论**：这是一种可接受的临时止血，但不是长期规范。未来必须把战略/资源/风险决策从 daily log 抽取为独立 decision artifact。

### 新技能4：任务追踪要以“三证”取代“口头推进”
**洞察**：稷下与轩辕任务卡都已把三证写成硬约束：产物路径 / Diff / 日志回执。  
**结论**：这是对“Agent 说做了，但无法验真”的结构性修复，值得扩展到全军团。
