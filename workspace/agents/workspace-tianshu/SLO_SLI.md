## 一，SRE框架概述

SRE（Site Reliability Engineering）将软件工程的严谨性引入运维，追求**可靠性 + 效率的平衡**。

天枢的SRE定位：**用SLO驱动运营决策，用错误预算平衡速度与质量。**

## 二，核心概念

| 概念 | 定义 | 天枢应用 |
|------|------|---------|
| **SLO** (Service Level Objective) | 目标可靠性水平 | 72h任务完成率 ≥ 95% |
| **SLI** (Service Level Indicator) | 实际可靠性指标 | 当前任务完成率 87% |
| **Error Budget** | 允许的不可靠额度 | 剩余5%空间 = 12h/季度 |
| **Toil** | 重复性手动劳动 | 战报填写 / 看板更新 |

### 3.2 Error Budget计算

```
Error Budget = 1 - SLO

季度总时间 = 2160小时 (90天)
任务闭环 SLO = 95% → Error Budget = 5% = 108小时允许失败时间

当前SLI = 87% → 已消耗 8% Error Budget = 172.8小时
剩余Error Budget = 3% = 64.8小时

⚠️  Error Budget消耗速度过快，需立即改善
```

## 四，SLI指标定义

### 4.1 任务闭环率 SLI

| 字段 | 定义 |
|------|------|
| **指标名称** | 72h任务闭环率 |
| **计算公式** | 已闭环任务数 / 总任务数 × 100% |
| **数据来源** | DASHBOARD.md 72h倒计时 |
| **刷新频率** | 每次任务状态变化 |
| **有效窗口** | 任务创建后72h内 |
| **排除项** | 外部依赖阻塞且有记录的 |

### 4.2 危机响应 SLI

| 字段 | 定义 |
|------|------|
| **指标名称** | P0危机响应时间 |
| **计算公式** | 触发时间 → 首次响应时间 |
| **数据来源** | HEARTBEAT.md 危机记录 |
| **刷新频率** | 每次危机 |
| **目标** | P0 ≤ 30min, P1 ≤ 2h, P2 ≤ 4h |

### 4.3 资源重配 SLI

| 字段 | 定义 |
|------|------|
| **指标名称** | 资源重配完成时间 |
| **计算公式** | 决策时间 → 资源到位时间 |
| **数据来源** | HEARTBEAT.md 资源调度记录 |
| **刷新频率** | 每次资源调整 |
| **目标** | ≤ 5min |

### 4.4 复盘完成率 SLI

| 字段 | 定义 |
|------|------|
| **指标名称** | 战役复盘完成率 |
| **计算公式** | 已复盘战役 / 需复盘战役 × 100% |
| **数据来源** | archive/reviews/ |
| **刷新频率** | 每周 |
| **目标** | 100%（强制） |

## 六，SLO告警规则

| 告警级别 | 触发条件 | 响应 | 负责人 |
|:--------:|---------|:----:|:------:|
| 🔴 **SLO Breach 紧急** | SLI < SLO - 5% | 立即介入 | 天枢 |
| 🟡 **Warning 警告** | SLI < SLO - 2% | 24h内优化 | 天枢 |
| ⚪ **Healthy 健康** | SLI ≥ SLO | 持续监控 | — |

**目标Toil自动化率**：Phase 4 达成 ≥ 80%

## 八，SLO与OKR联动

```
SLO达成 → Error Budget释放 → 可用于创新实验 → OKR突破
    ↑
SLO失败 → Error Budget消耗 → 需紧急修复 → OKR回撤
```

| SLO | 影响OKR | 联动方向 |
|-----|---------|---------|
| 任务闭环率 | O1-KR4 | 正向 |
| 危机响应 | O1-KR3 | 正向 |
| 复盘完成率 | O2-KR2 | 正向 |

## 九，SLO季度评估流程

```
每季度末(3月/6月/9月/12月)：
    ↓
1. 收集所有SLI数据
2. 计算Error Budget消耗
3. 评估各SLO达成状态
4. 调整下季度SLO阈值
5. 制定SLO改进计划
6. 更新本文件
```
