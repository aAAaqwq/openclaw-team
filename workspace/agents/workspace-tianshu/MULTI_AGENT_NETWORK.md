## 二，Agent通信协议

### 2.1 消息格式

```json
{
  "from": "轩辕",
  "to": ["天枢", "凤凰"],
  "type": "task_complete",
  "task_id": "T-2026-0503-001",
  "payload": {
    "status": "success",
    "output": "...",
    "duration": "2.5h",
    "issues": [],
    "next_steps": ["handoff_to_phoenix"]
  },
  "timestamp": "2026-05-03T14:00:00+08:00",
  "priority": "normal"
}
```

### 2.2 消息类型

| 类型 | 含义 | 方向 | 优先级 |
|:----:|------|:----:|:------:|
| `task_assign` | 任务分配 | Hub→Spoke | medium |
| `task_complete` | 任务完成 | Spoke→Hub | medium |
| `progress_update` | 进度更新 | Spoke→Hub | low |
| `blocked` | 阻塞报告 | Spoke→Hub | **high** |
| `resource_request` | 资源请求 | Spoke→Hub | **high** |
| `info_share` | 信息共享 | Spoke↔Spoke | low |
| `peer_request` | 跨域协作请求 | Spoke→Spoke | medium |
| `quality_gate` | 质量门禁 | Hub→Spoke | high |
| `escalation` | 升级上报 | Spoke→Hub→Kunlun | **critical** |
| `broadcast` | 全局广播 | Hub→All | variable |

## 三，工作流编排

### 3.1 标准协作流程

```
天枢：战役规划 → 任务分解 → RACI分配
    ↓
轩辕：技术评估 → 工作量估算 → 排期
    ↓
凤凰：内容评估 → 创意方案 → 排期
    ↓
鲲鹏：获客策略 → 渠道分配 → 排期
    ↓
烛龙：预算确认 → 现金流规划 → 风险提示
    ↓
天枢：统一调度 → 资源分配 → 启动战役
    ↓
轩辕+凤凰+鲲鹏+烛龙：并行执行 → 进展汇报
    ↓
天枢：进度监控 + 资源动态调整 + 阻塞处理
    ↓
战役闭环 → review
```

### 3.2 协作SLA

| 协作类型 | SLA | 超时处理 |
|---------|:----:|---------|
| 天枢调度→响应 | 15min | 自动提醒+升级 |
| 任务分配→确认 | 30min | 二次分配 |
| 阻塞报告→响应 | 5min | 自动升级 |
| 跨域协作→首次响应 | 1h | 天枢介入 |
| 质量门禁→审核 | 2h | 自动延时 |

## 四，自主决策权限

### 4.1 各Agent自主决策范围

| Agent | 自主决策 | 需天枢确认 | 需Kunlun/创始人 |
|-------|---------|:----------:|:-------------:|
| **轩辕** | 技术方案选择、技术栈、架构决策 | 重大架构变更 | 极重要 |
| **凤凰** | 内容主题、发布时间、渠道选择 | 品牌相关 | 战略级内容 |
| **鲲鹏** | 获客渠道、预算内投放优化 | 预算调整>20% | 新市场开拓 |
| **烛龙** | 日常现金流管理、战报生成 | 重大支出 | 融资相关 |

### 4.2 决策升级路径

```
Agent自主决策
    ↑
├── 常规决策 → 直接执行
├── 非常规但安全 → 通知天枢
└── 非常规且有风险 → 暂停 → 申请天枢批准
                           ↓
                  天枢评估
                    ↓
            ├── 天枢可决策 → 下达指令
            └── 超越权限 → 升级到Kunlun/创始人
```

## 五，冲突解决机制

### 5.1 资源冲突

```
场景：两个Agent同时需要相同资源

解决：
1. 天枢评估优先级
2. 高优先级获得资源
3. 低优先级调整排期
4. 记录冲突到DASHBOARD
```

### 5.2 方案冲突

```
场景：轩辕和凤凰对实施路线有分歧

解决：
1. 各自提交可行性分析
2. 天枢做成本/收益/风险评估
3. 天枢仲裁决策
4. 记录决策理由
```

### 5.3 优先级冲突

```
场景：多个任务争抢调度顺序

解决：
1. 紧急度矩阵 (紧急性×重要性)
2. 依赖关系分析
3. 资源约束检查
4. 天枢最终裁定
```

## 七，演化路线

### 7.1 当前（v1.0）

```
Hub-and-Spoke 拓扑，天枢作为中央调度
所有跨域协作经过天枢
Agent自主决策比例：约40%
```

### 7.2 中期（v2.0）

```
Hub + 部分Peer-to-Peer
常见协作模式直连
Agent自主决策比例：约60%
天枢主要做仲裁和非常规决策
```

### 7.3 终极（v3.0）

```
完全连通 + 自治网络
Agent可自主协商资源
Agent自主决策比例：约85%
天枢做战略+监管+赋能
```
