## 四、失败模式

### 失败模式 1：证据链偏“文档化”，偏离可执行验证
今天大量产物是高质量文档与 skill，但缺少以下硬证据：
- 测试报告
- 编译/构建结果
- lint/typecheck 输出
- git commit / diff 级证据

**根因**：
- 当前阶段以工作空间初始化和能力建设为主
- 尚未建立 nightly 自动采集命令
- 还没有把“证据”纳入交付门禁

### 失败模式 2：Git 状态不可用于有效增量审计
当前 `git status` 显示大范围未跟踪目录，说明仓库边界或版本管理策略尚未收敛。

**影响**：
- 难以精确抽取“今日实际改动”
- nightly 复盘只能依赖 mtime 和文件存在性
- 后续审计、回滚、归因成本上升

**根因**：
- 工作空间初始化后未完成规范提交
- agent workspace 与上层 workspace 的版本边界仍较粗糙

### 失败模式 3：记忆检索链路失效
`memory_search` 返回不可用（database is not open），说明语义记忆检索链路当前失效。

**影响**：
- 无法稳定回收历史决策与上下文
- 影响复盘完整性与长期连续性

**根因**：
- embedding/provider 或 memory 数据库状态异常

### 失败模式 4：夜间复盘仍依赖人工解释
虽然已有 `HEARTBEAT.md`、`EVOLUTION.md`，但夜间《暗夜熔炉》尚未形成自动汇聚：
- 当日新增文件
- 测试结果
- 异常日志
- 明日动作项

仍需人工拼装。

## 六、工程最优解判断

如果从“今天最值得保留什么”来判断，答案不是某个单文件，而是下面这条路线：

> **以 skill 为能力单元，以 memory 为长期记忆，以 heartbeat 为巡检触发，以 nightly-forge 为复盘闭环。**

这是当前最优工程解，因为它兼顾：
- **可复用**：经验能反复调用
- **可审计**：产物能留下证据
- **可扩展**：后续可接入测试/构建/日志系统
- **可并行**：适合多 Agent 协作

当前短板不是方向错误，而是**自动证据化程度不够**。下一步不该继续堆文档，而该把这些制度接上线：
- 自动检测
- 自动记录
- 自动归档
- 自动复盘
