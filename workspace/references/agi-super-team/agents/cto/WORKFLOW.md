## 📋 Standard Workflows (from team library)

### Code Review

# 代码审查 — Code Review

## 元信息
- **触发条件**: 每次 git commit / PR 提交后
- **执行频率**: 每次提交
- **预计耗时**: 5-15 分钟/提交
- **负责 Agent**: PE
- **协作 Agent**: 相关 Agent（功能验收）

## 执行步骤

### Step 1: 快速扫描 (3 min)
```
## Code Review: [commit hash / PR title]

### 概览
| 维度 | 检查 |
|------|------|
| 文件变更 | X 个文件, +XXX/-XXX 行 |
| 影响范围 | [哪些模块] |
| 风险等级 | 🟢 低 / 🟡 中 / 🔴 高 |
```

### Step 2: 实质性审查 (5-10 min)

**必须检查:**
1. **功能正确性**: 代码做了它该做的事吗？
2. **数据结构**: 数据怎么流动？结构对吗？
3. **边界条件**: 空输入、大数据量、并发冲突
4. **安全**: 有没有硬编码密钥、SQL 注入、权限问题
5. **依赖**: 新增依赖是否必要？版本是否锁定？

**检查清单:**
- [ ] 无硬编码密钥/路径
- [ ] 错误处理覆盖异常路径
- [ ] 无全局状态泄漏
- [ ] 函数长度 < 50 行
- [ ] 变量命名自解释

### Step 3: 决策 (2 min)
- **✅ Approve**: 代码质量可接受，可以合并
- **⚠️ Request Changes**: 有具体修改建议，列出需改项
- **❌ Block**: 有严重问题（安全/正确性），必须修复

## 输出格式
```
## Review 结论: ✅ Approve / ⚠️ Changes / ❌ Block

### 问题
1. [L42] [问题描述] → [建议修改]

### 肯定
- [值得表扬的点]
```

## 异常处理
- 提交规模过大（> 500 行）→ 要求拆分
- 涉及敏感文件（config/密钥）→ 最高优先级审查

---

*"Talk is cheap. Show me the code. 但审查代码时，我会先 show you the problems."*


### Deploy Pipeline

# 部署流水线 — Deploy Pipeline

## 元信息
- **触发条件**: 功能开发完成 + CPO 验收通过
- **执行频率**: 按需
- **预计耗时**: 15-30 分钟
- **负责 Agent**: PE
- **协作 Agent**: COO（基础设施）、CPPO（验收）

## 部署流程

### Phase 1: 部署前检查 (5 min)
```
## 部署前检查清单

- [ ] 所有测试通过
- [ ] 无硬编码密钥/路径
- [ ] 环境变量配置确认
- [ ] 数据库迁移脚本准备
- [ ] 回滚方案准备
- [ ] 监控告警配置
```

### Phase 2: 部署执行 (5-15 min)
```
1. 创建部署分支: deploy/YYYY-MM-DD-[feature]
2. 合并到 main/master
3. 执行部署命令
4. 验证服务启动正常
5. 执行冒烟测试（关键路径）
```

### Phase 3: 部署后验证 (5 min)
- 核心功能冒烟测试
- 监控面板检查（错误率、延迟）
- 日志检查（无异常报错）

### Phase 4: 观察期 (30 min)
- 部署后 30 分钟内保持监控
- 无异常 → 标记部署成功
- 有异常 → 立即回滚

## 回滚流程
```
1. 通知 COO + CEO
2. 执行回滚到上一稳定版本
3. 确认服务恢复
4. 记录事故 → 根因分析
```

## 输出
- 部署记录 → git tag
- 异常 → 立即告警

---

*"能回滚的部署才是安全的部署。永远准备好退路。"*


### Performance Profiling

# 性能剖析 — Performance Profiling

## 元信息
- **触发条件**: 用户反馈延迟 / 监控告警 / 定期优化
- **执行频率**: 按需（至少每月 1 次）
- **预计耗时**: 1-2 小时
- **负责 Agent**: PE

## 执行步骤

### Step 1: 定义基准 (10 min)
```
## 性能基准

| 指标 | 当前值 | 目标值 | 测量方法 |
|------|--------|--------|----------|
| API 响应 P50 | 230ms | < 200ms | curl -w |
| API 响应 P99 | 1.2s | < 500ms | 压测 |
| 内存使用 | 450MB | < 500MB | /proc |
| 启动时间 | 8s | < 5s | time |
```

### Step 2: Profiling (20-60 min)
1. **CPU 热点**: 找到最耗 CPU 的函数
2. **内存分析**: 找到内存泄漏/大量分配
3. **I/O 瓶颈**: 磁盘/网络 I/O 等待
4. **并发分析**: 锁竞争/线程阻塞

### Step 3: 优化实施 (20-40 min)
按 ROI 排序优化：
```
| 优化项 | 预期提升 | 修复成本 | ROI |
|--------|----------|----------|-----|
| [具体优化] | X% | Xh | 高 |
```

**优化顺序:**
1. 算法/数据结构优化（最高 ROI）
2. 缓存（重复计算 → 缓存）
3. 并发（串行 → 并行）
4. I/O（批量 → 减少调用）
5. 语言层（编译优化/JIT）

### Step 4: 验证 (10 min)
- 对比优化前后基准
- 确认无功能回归
- 更新基准文档

## 原则
- **先 Profile 再优化** — 不猜测瓶颈
- **量化验证** — 优化前后必须有数据对比
- **不过度优化** — 达到目标值就停

## 输出
- 性能报告 → CEO
- 优化 PR → 代码库

---

*"Premature optimization is the root of all evil. 但 Data-driven optimization is the root of all performance."*


### Tech Debt Audit

# 技术债审计 — Tech Debt Audit

## 元信息
- **触发条件**: 每周日 15:00
- **执行频率**: 每周
- **预计耗时**: 20 分钟
- **负责 Agent**: PE

## 执行步骤

### Step 1: 债务盘点 (10 min)
扫描所有项目的技术债：

```
## 技术债周报 (YYYY-WW)

### 债务清单
| # | 项目 | 类型 | 描述 | 严重度 | 修复成本 |
|---|------|------|------|--------|----------|
| 1 | MediaClaw | 硬编码 | API URL 硬编码 | 🔴 高 | 30min |
| 2 | super-quant | TODO | 未处理的边界条件 | 🟡 中 | 1h |
| 3 | workspace | 过期依赖 | v2.x → v3.x | 🟡 中 | 2h |
```

**债务类型:**
- 硬编码/魔法数字
- 未处理的 TODO（> 1 周）
- 过期依赖
- 缺少测试
- 代码重复
- 性能瓶颈
- 文档缺失

### Step 2: 优先级排序 (5 min)
按 `严重度 × 影响范围 / 修复成本` 排序

### Step 3: 制定偿还计划 (5 min)
```
## 本周偿还计划
| 债务 | 预计时间 | 优先级 |
|------|----------|--------|
| #1 | 周一 30min | P1 |
| #3 | 周三 2h | P2 |
```

## 原则
- **每周至少还 1 笔技术债**，不让债务滚雪球
- **P0 债务当日还**（安全/正确性问题）
- **P1 债务本周还**（影响效率的问题）
- **P2 债务排期还**（优化类）

## 输出
- 技术债报告 → CEO
- 本周偿还任务 → 加入日常计划

---

*"技术债就像信用卡：借了就得还，不还利息比本金更贵。每周还一点，永远不欠债。"*

### Crisis Escalation

# 危机升级 — Crisis Escalation

## 元信息
- **触发条件**: 以下任一条件满足时触发
- **执行频率**: 按需
- **预计耗时**: 视严重程度 5-60 分钟
- **负责 Agent**: 发现者 → CEO → 创始人

## 触发条件

| 级别 | 定义 | 示例 | 响应时间 |
|------|------|------|----------|
| 🔴 P0-Critical | 影响收入/安全/核心功能 | API key 泄露、服务宕机、资金异常 | 5 分钟内 |
| 🟠 P1-Urgent | 影响效率/用户体验 | 关键 Agent 无响应、数据管线中断 | 30 分钟内 |
| 🟡 P2-Warning | 需要关注但可容忍 | 成本超标、性能下降、外部依赖异常 | 24 小时内 |

## 执行步骤

### Step 1: 发现与定性 (0-2 分钟)
发现者（任意 Agent）执行：
1. 确认问题真实性（排除误报）
2. 判定严重等级（P0/P1/P2）
3. 按以下格式上报：

```
🚨 [P0/P1/P2] 问题简述
📍 影响范围: <哪些模块/功能>
⚡ 当前状态: <正在发生/已发生/有风险>
🛠 初步处理: <已做的>
⏰ 需要支援: <找谁>
```

### Step 2: 升级路径

```
P2 → 相关 C-Suite Agent 自行处理，CEO 抄送
P1 → 相关 C-Suite + CEO 协调，30 分钟内响应
P0 → CEO → 创始人，5 分钟内响应
    → 安全问题: 通知 CLO
    → 财务问题: 通知 CFO
    → 技术问题: 通知 PE + COO
    → 数据问题: 通知 CDO
```

### Step 3: 处理与止损
1. **P0**: 先止损，后修复。恢复服务 > 根因分析 > 预防
2. **P1**: 并行处理（一人止损，一人查因）
3. **P2**: 纳入待办，下一个工作时间窗口处理

### Step 4: 复盘与闭环
P0/P1 处理完成后：
1. 写事故报告（根因、影响、处理、预防）
2. 更新相关 SOUL.md / MEMORY.md
3. P0 必须 24h 内出复盘，P1 必须 48h 内

## 输出
- **即时**: 升级消息（群/私聊）
- **事后**: 事故报告 → `memory/` 目录
- **预防**: 流程/SOP 更新（如需要）

## 质量门禁
- [ ] P0 5 分钟内开始响应
- [ ] P0 30 分钟内恢复或降级
- [ ] 事故报告在 SLA 内产出
- [ ] 至少 1 个预防措施落地

## 异常处理
- CEO 无响应（P0）→ 直接通知创始人
- 问题范围持续扩大 → 召集跨部门紧急会议
- 无法在 SLA 内解决 → 降级方案 + 沟通预期

---

*"P0 先止血，P1 先隔离，P2 排队处理。永远不会等到'准备好'才行动。"*


### Cross Agent Handoff

# 跨 Agent 交接 — Cross-Agent Handoff

## 元信息
- **触发条件**: 任务需要从 Agent A 移交给 Agent B
- **执行频率**: 按需
- **预计耗时**: 3 分钟
- **负责 Agent**: 发起方

## 输入
- 当前任务状态（完成了什么，还差什么）
- 下一步需要做什么（具体，不是"继续做"）
- 依赖的文件/数据/凭证

## 执行步骤

### Step 1: 发起交接
Agent A 通过 `sessions_send` 发送交接消息：

```markdown
## 📋 任务交接: [任务名]

**来源**: [Agent A] → **目标**: [Agent B]
**背景**: [一句话说明为什么交接]

### ✅ 已完成
- [具体产出 1]
- [具体产出 2]

### 🎯 待完成
- [具体待做 1，含验收标准]
- [具体待做 2]

### 📁 关键文件
- `path/to/file1` — 用途说明
- `path/to/file2` — 用途说明

### ⚠️ 注意事项
- [坑/坑/坑]

### 📊 截止时间
- [日期 + 时间]
```

### Step 2: 接收确认
Agent B 收到后 10 分钟内回复：
```
✅ 已接收任务 [任务名]
📋 预计完成: [时间]
❓ 有以下疑问: [0-N 个，没有则省略]
```

### Step 3: 完成确认
Agent B 完成后通知 Agent A：
```
✅ 任务 [任务名] 已完成
📄 产出: [路径/链接]
📊 结果: [关键数据]
```

## 交接质量标准
- **绝不交空手** — 必须有具体产出（文件/代码/数据），不能只说"我做了调研"
- **绝不交模糊** — 待完成项必须有验收标准，不能说"继续优化"
- **绝不交无主** — 必须明确截止时间，不能说"有空再做"
- **绝不交黑盒** — 注意事项必须写清楚踩过的坑，不能让接手人重新踩

## 异常处理
- 接收方 10 分钟无回复 → 发起方通知 CEO
- 接收方无法独立完成 → 反向交接回发起方或请求 CEO 协调
- 任务在交接后卡住 → 最后一任负责方承担，不得反向推诿

---

*"好的交接 = 接手者不需要问任何问题就能开始干活。"*


### Daily Standup

# 每日站会 — Daily Standup

## 元信息
- **触发条件**: 每日定时 (08:00 CST) 或 CEO 手动触发
- **执行频率**: 每天
- **预计耗时**: 5 分钟/Agent
- **负责 Agent**: CEO（组织） + 各 C-Suite（汇报）
- **协作 Agent**: 全员

## 输入
- 前一日产出记录（各 agent `memory/` 目录）
- 当前进行中的任务列表
- 阻塞/依赖项

## 执行步骤

### Phase 1: CEO 收集 (08:00)
1. 遍历各 agent 的 `memory/` 最近记录
2. 检查 cron 任务执行状态（`openclaw cron list`）
3. 汇总异常/超时任务

### Phase 2: 各 Agent 汇报 (08:05)
每个 Agent 回报三件事（≤ 3 句话）：

```
📊 [角色] 昨日完成: <1件核心产出>
🎯 [角色] 今日计划: <1件核心目标>
⚠️ [角色] 阻塞/风险: <0-1个，无则省略>
```

**示例:**
```
📊 CCO 昨日完成: 3篇小红书笔记发布，1篇过500阅读
🎯 CCO 今日计划: 拆解抖音爆款5条，产出1个脚本模板
```

### Phase 3: CEO 决策 (08:15)
1. 确认优先级调整（有变更则通知相关 Agent）
2. 分配跨部门任务（指定负责人 + 截止时间）
3. 标记 P0 阻塞项（立即处理）

## 输出
- **群消息**: 站会摘要（≤ 10 行）
- **文件**: `memory/YYYY-MM-DD.md` 更新各 agent 进展

## 质量门禁
- [ ] 每个 Agent 至少报告了昨日完成和今日计划
- [ ] P0 阻塞项有明确负责人和截止时间
- [ ] 站会总时长 ≤ 15 分钟

## 异常处理
- Agent 无响应 → CEO 标记为「静默」，连续 2 天启动健康检查
- P0 冲突（多个 P0 抢资源）→ CEO 裁决优先级
- 跨部门依赖卡住 → CEO 指定协调人 + 2h SLA

---

*"站会不是汇报会，是对齐会。15 分钟解决所有信息不对称。"*


### Weekly Review

# 周度复盘 — Weekly Review

## 元信息
- **触发条件**: 每周日 20:00 CST
- **执行频率**: 每周
- **预计耗时**: 30 分钟
- **负责 Agent**: CEO
- **协作 Agent**: 全员（各 Agent 提交周报）

## 输入
- 本周各 agent `memory/` 全部记录
- 周初设定的目标列表
- 本周所有 P0/P1 任务完成情况

## 执行步骤

### Phase 1: 数据收集 (周日 20:00)
1. CEO 拉取本周所有 agent 的 git commits、任务完成记录
2. 统计核心指标：
   - 代码提交数、内容产出数、数据采集量
   - P0/P1 完成率
   - Token/成本消耗

### Phase 2: 各 Agent 周报 (周日 20:10)
每个 Agent 提交（≤ 200 字）：

```
📋 [角色] 周报
✅ 完成: (数字) 件核心产出
❌ 未完成: (哪些 + 原因)
📈 数据: (关键数字)
💡 下周重点: (1-3 件)
🔧 改进: (本周学到的)
```

### Phase 3: CEO 综合评估 (周日 20:25)
1. **目标达成率** — 本周 OKR 完成百分比
2. **效率分析** — 哪些 Agent 高效/低效，原因分析
3. **资源瓶颈** — 是否需要新增 skill / 调整分工
4. **风险预警** — 下周可能出问题的事项

### Phase 4: 下周规划 (周日 20:30)
1. CEO 设定下周 P0/P1
2. 分配跨部门项目
3. 更新团队看板

## 输出
- **群消息**: 周报摘要 + 下周重点（≤ 20 行）
- **文件**: `memory/YYYY-MM-DD.md` 周复盘记录

## 质量门禁
- [ ] 所有 Agent 提交周报
- [ ] P0 完成率 ≥ 80%
- [ ] 下周 P0 明确且 ≤ 3 个
- [ ] 有至少 1 个改进措施落地

## 异常处理
- Agent 周报缺失 → 记录为「周度沉默」，连续 2 周升级为 P1
- P0 完成率 < 50% → CEO 重新评估可行性，调整目标或增派资源
- 跨部门项目连续 2 周无进展 → 取消或降级

---

*"不复盘的工作等于没做。数据说话，数字不撒谎。"*
