# WORKFLOW.md — Simons 量化交易工作流

_信号驱动 · 纪律执行 · 持续迭代_

## 交易闭环

```
信号扫描 → 策略匹配 → 风险评估 → 执行 → 复盘
    ↑                                    |
    └──────────── 模型迭代 ←──────────────┘
```

## Phase 1: 信号扫描

- **数据源**: 加密市场(BTC/ETH)、Polymarket、A股、新闻情绪
- **频率**: 加密实时，Polymarket 4h，A股盘中
- **输出**: 信号列表（方向、强度、来源、时间戳）

## Phase 2: 策略匹配

- 将信号匹配到已有策略池
- 无匹配策略 → 研发阶段（不直接交易）
- 验证清单：[ ] 样本外测试 [ ] 扣除交易成本 [ ] 相关性检查

## Phase 3: 风险评估

- 单笔最大风险: ≤ 2% 总资金
- 组合总风险: ≤ 6% 总资金
- 相关性检查：新头寸与现有头寸的相关系数
- 止损设定：数学定义，非主观判断

## Phase 4: 执行

- 机械执行，不改计划
- 执行偏差记录（滑点、延迟、心理干扰）
- 异常立即上报 CEO

## Phase 5: 复盘（每日/每周）

- 每笔交易 post-mortem（尤其是亏损）
- 策略表现：Sharpe、最大回撤、胜率
- 错误分类：信号错误 / 执行偏差 / 系统故障
- 模型迭代建议

## 异常处理

| 级别 | 触发 | 动作 |
|------|------|------|
| P0 | 单笔亏损 > 5% 或系统故障 | 立即停止交易，上报 CEO |
| P1 | 策略连续 3 次亏损 | 暂停策略，启动复盘 |
| P2 | 信号质量下降 | 降频交易，加强过滤 |

### Market Morning Brief

# 市场晨报 — Market Morning Brief

## 元信息
- **触发条件**: 每日 08:00 CST（美股开盘前）
- **执行频率**: 每天（工作日）
- **预计耗时**: 10 分钟
- **负责 Agent**: CQO

## 执行步骤

### Step 1: 数据采集 (5 min)
```
## 市场晨报 (YYYY-MM-DD)

### 宏观指标
| 指标 | 最新值 | 变化 | 信号 |
|------|--------|------|------|
| BTC/USD | $XX,XXX | +/-X% | |
| ETH/USD | $X,XXX | +/-X% | |
| 恐惧贪婪指数 | XX | +/-X | |

### 关键事件
- [事件1]: [影响分析]
- [事件2]: [影响分析]

### 量化策略状态
| 策略 | 状态 | 今日信号 | 持仓 |
|------|------|----------|------|
| Turtle | RUNNING | 无信号 | 空 |
| MeanRev | RUNNING | RSI=28 → 买入 | BTC |
```

### Step 2: 信号判断 (3 min)
- 有新信号？→ 列出具体参数和操作建议
- 有异常波动？→ 标记风险等级
- 策略需要调整？→ 列出建议参数变更

### Step 3: 操作决策 (2 min)
- **今日操作**: [具体买入/卖出/持有 + 理由]
- **风险提醒**: [止损位/仓位控制]
- **关注项**: [今天要盯什么]

## 输出
- 晨报 → CEO + 群（≤ 15 行）
- 交易信号 → 执行（如有）

## 异常处理
- 数据源不可用 → 切换备用数据源，延迟 30 分钟发布
- 市场剧烈波动（>5%）→ 🔴 升级为 P1，通知 CEO

---

*"盘前 5 分钟的决策，比盘中 5 小时的焦虑更值钱。"*


### Strategy Backtest

# 策略回测 — Strategy Backtest

## 元信息
- **触发条件**: 每周日 14:00 + 策略修改后
- **执行频率**: 每周 + 按需
- **预计耗时**: 30 分钟
- **负责 Agent**: CQO

## 执行步骤

### Step 1: 数据准备 (5 min)
- 确认回测数据完整性（无缺失/异常）
- 设定回测参数：时间范围、初始资金、手续费

### Step 2: 执行回测 (10 min)
对每个活跃策略执行回测：
```
## 回测报告: [策略名] (过去 7/30/90 天)

### 核心指标
| 指标 | 7天 | 30天 | 90天 |
|------|-----|------|------|
| 收益率 | X% | X% | X% |
| 最大回撤 | X% | X% | X% |
| 夏普比率 | X.X | X.X | X.X |
| 胜率 | X% | X% | X% |
| 交易次数 | X | X | X |
```

### Step 3: 对比分析 (10 min)
- 策略 vs 持有（Buy & Hold）
- 策略 vs 上期回测（是否退化）
- 多策略相关性（是否分散）

### Step 4: 决策 (5 min)
- **继续运行**: 指标正常
- **参数调优**: 特定指标下降 → 列出建议调整
- **暂停策略**: 连续亏损 + 回撤超限
- **新策略评估**: 有新想法 → 加入回测队列

## 输出
- 回测报告 → CEO
- 参数变更建议（如有）→ CTO-Dev 执行
- 策略启停决策

---

*"不回测的策略等于赌博。数据证明有效，才值得真金白银。"*

## 🔄 Shared Workflows

### Crisis Escalation

# 危机升级 — Crisis Escalation

## 元信息
- **触发条件**: 以下任一条件满足时触发
- **执行频率**: 按需
- **预计耗时**: 视严重程度 5-60 分钟
- **负责 Agent**: 发现者 → CEO → 创始人

## 触发条件

| 级别 | 定义 | 示例 | 响应时间 |
|------|------|------|----------|
| 🔴 P0-Critical | 影响收入/安全/核心功能 | API key 泄露、服务宕机、资金异常 | 5 分钟内 |
| 🟠 P1-Urgent | 影响效率/用户体验 | 关键 Agent 无响应、数据管线中断 | 30 分钟内 |
| 🟡 P2-Warning | 需要关注但可容忍 | 成本超标、性能下降、外部依赖异常 | 24 小时内 |

## 执行步骤

### Step 1: 发现与定性 (0-2 分钟)
发现者（任意 Agent）执行：
1. 确认问题真实性（排除误报）
2. 判定严重等级（P0/P1/P2）
3. 按以下格式上报：

```
🚨 [P0/P1/P2] 问题简述
📍 影响范围: <哪些模块/功能>
⚡ 当前状态: <正在发生/已发生/有风险>
🛠 初步处理: <已做的>
⏰ 需要支援: <找谁>
```

### Step 2: 升级路径

```
P2 → 相关 C-Suite Agent 自行处理，CEO 抄送
P1 → 相关 C-Suite + CEO 协调，30 分钟内响应
P0 → CEO → 创始人，5 分钟内响应
    → 安全问题: 通知 CLO
    → 财务问题: 通知 CFO
    → 技术问题: 通知 PE + COO
    → 数据问题: 通知 CDO
```

### Step 3: 处理与止损
1. **P0**: 先止损，后修复。恢复服务 > 根因分析 > 预防
2. **P1**: 并行处理（一人止损，一人查因）
3. **P2**: 纳入待办，下一个工作时间窗口处理

### Step 4: 复盘与闭环
P0/P1 处理完成后：
1. 写事故报告（根因、影响、处理、预防）
2. 更新相关 SOUL.md / MEMORY.md
3. P0 必须 24h 内出复盘，P1 必须 48h 内

## 输出
- **即时**: 升级消息（群/私聊）
- **事后**: 事故报告 → `memory/` 目录
- **预防**: 流程/SOP 更新（如需要）

## 质量门禁
- [ ] P0 5 分钟内开始响应
- [ ] P0 30 分钟内恢复或降级
- [ ] 事故报告在 SLA 内产出
- [ ] 至少 1 个预防措施落地

## 异常处理
- CEO 无响应（P0）→ 直接通知创始人
- 问题范围持续扩大 → 召集跨部门紧急会议
- 无法在 SLA 内解决 → 降级方案 + 沟通预期

---

*"P0 先止血，P1 先隔离，P2 排队处理。永远不会等到'准备好'才行动。"*


### Cross Agent Handoff

# 跨 Agent 交接 — Cross-Agent Handoff

## 元信息
- **触发条件**: 任务需要从 Agent A 移交给 Agent B
- **执行频率**: 按需
- **预计耗时**: 3 分钟
- **负责 Agent**: 发起方

## 输入
- 当前任务状态（完成了什么，还差什么）
- 下一步需要做什么（具体，不是"继续做"）
- 依赖的文件/数据/凭证

## 执行步骤

### Step 1: 发起交接
Agent A 通过 `sessions_send` 发送交接消息：

```markdown
## 📋 任务交接: [任务名]

**来源**: [Agent A] → **目标**: [Agent B]
**背景**: [一句话说明为什么交接]

### ✅ 已完成
- [具体产出 1]
- [具体产出 2]

### 🎯 待完成
- [具体待做 1，含验收标准]
- [具体待做 2]

### 📁 关键文件
- `path/to/file1` — 用途说明
- `path/to/file2` — 用途说明

### ⚠️ 注意事项
- [坑/坑/坑]

### 📊 截止时间
- [日期 + 时间]
```

### Step 2: 接收确认
Agent B 收到后 10 分钟内回复：
```
✅ 已接收任务 [任务名]
📋 预计完成: [时间]
❓ 有以下疑问: [0-N 个，没有则省略]
```

### Step 3: 完成确认
Agent B 完成后通知 Agent A：
```
✅ 任务 [任务名] 已完成
📄 产出: [路径/链接]
📊 结果: [关键数据]
```

## 交接质量标准
- **绝不交空手** — 必须有具体产出（文件/代码/数据），不能只说"我做了调研"
- **绝不交模糊** — 待完成项必须有验收标准，不能说"继续优化"
- **绝不交无主** — 必须明确截止时间，不能说"有空再做"
- **绝不交黑盒** — 注意事项必须写清楚踩过的坑，不能让接手人重新踩

## 异常处理
- 接收方 10 分钟无回复 → 发起方通知 CEO
- 接收方无法独立完成 → 反向交接回发起方或请求 CEO 协调
- 任务在交接后卡住 → 最后一任负责方承担，不得反向推诿

---

*"好的交接 = 接手者不需要问任何问题就能开始干活。"*


### Daily Standup

# 每日站会 — Daily Standup

## 元信息
- **触发条件**: 每日定时 (08:00 CST) 或 CEO 手动触发
- **执行频率**: 每天
- **预计耗时**: 5 分钟/Agent
- **负责 Agent**: CEO（组织） + 各 C-Suite（汇报）
- **协作 Agent**: 全员

## 输入
- 前一日产出记录（各 agent `memory/` 目录）
- 当前进行中的任务列表
- 阻塞/依赖项

## 执行步骤

### Phase 1: CEO 收集 (08:00)
1. 遍历各 agent 的 `memory/` 最近记录
2. 检查 cron 任务执行状态（`openclaw cron list`）
3. 汇总异常/超时任务

### Phase 2: 各 Agent 汇报 (08:05)
每个 Agent 回报三件事（≤ 3 句话）：

```
📊 [角色] 昨日完成: <1件核心产出>
🎯 [角色] 今日计划: <1件核心目标>
⚠️ [角色] 阻塞/风险: <0-1个，无则省略>
```

**示例:**
```
📊 CCO 昨日完成: 3篇小红书笔记发布，1篇过500阅读
🎯 CCO 今日计划: 拆解抖音爆款5条，产出1个脚本模板
```

### Phase 3: CEO 决策 (08:15)
1. 确认优先级调整（有变更则通知相关 Agent）
2. 分配跨部门任务（指定负责人 + 截止时间）
3. 标记 P0 阻塞项（立即处理）

## 输出
- **群消息**: 站会摘要（≤ 10 行）
- **文件**: `memory/YYYY-MM-DD.md` 更新各 agent 进展

## 质量门禁
- [ ] 每个 Agent 至少报告了昨日完成和今日计划
- [ ] P0 阻塞项有明确负责人和截止时间
- [ ] 站会总时长 ≤ 15 分钟

## 异常处理
- Agent 无响应 → CEO 标记为「静默」，连续 2 天启动健康检查
- P0 冲突（多个 P0 抢资源）→ CEO 裁决优先级
- 跨部门依赖卡住 → CEO 指定协调人 + 2h SLA

---

*"站会不是汇报会，是对齐会。15 分钟解决所有信息不对称。"*


### Weekly Review

# 周度复盘 — Weekly Review

## 元信息
- **触发条件**: 每周日 20:00 CST
- **执行频率**: 每周
- **预计耗时**: 30 分钟
- **负责 Agent**: CEO
- **协作 Agent**: 全员（各 Agent 提交周报）

## 输入
- 本周各 agent `memory/` 全部记录
- 周初设定的目标列表
- 本周所有 P0/P1 任务完成情况

## 执行步骤

### Phase 1: 数据收集 (周日 20:00)
1. CEO 拉取本周所有 agent 的 git commits、任务完成记录
2. 统计核心指标：
   - 代码提交数、内容产出数、数据采集量
   - P0/P1 完成率
   - Token/成本消耗

### Phase 2: 各 Agent 周报 (周日 20:10)
每个 Agent 提交（≤ 200 字）：

```
📋 [角色] 周报
✅ 完成: (数字) 件核心产出
❌ 未完成: (哪些 + 原因)
📈 数据: (关键数字)
💡 下周重点: (1-3 件)
🔧 改进: (本周学到的)
```

### Phase 3: CEO 综合评估 (周日 20:25)
1. **目标达成率** — 本周 OKR 完成百分比
2. **效率分析** — 哪些 Agent 高效/低效，原因分析
3. **资源瓶颈** — 是否需要新增 skill / 调整分工
4. **风险预警** — 下周可能出问题的事项

### Phase 4: 下周规划 (周日 20:30)
1. CEO 设定下周 P0/P1
2. 分配跨部门项目
3. 更新团队看板

## 输出
- **群消息**: 周报摘要 + 下周重点（≤ 20 行）
- **文件**: `memory/YYYY-MM-DD.md` 周复盘记录

## 质量门禁
- [ ] 所有 Agent 提交周报
- [ ] P0 完成率 ≥ 80%
- [ ] 下周 P0 明确且 ≤ 3 个
- [ ] 有至少 1 个改进措施落地

## 异常处理
- Agent 周报缺失 → 记录为「周度沉默」，连续 2 周升级为 P1
- P0 完成率 < 50% → CEO 重新评估可行性，调整目标或增派资源
- 跨部门项目连续 2 周无进展 → 取消或降级

---

*"不复盘的工作等于没做。数据说话，数字不撒谎。"*
