# WORKFLOW.md — COO Standard Workflows

## 📋 Standard Workflows (from team library)

### Incident Response

# 故障响应 — Incident Response

## 元信息
- **触发条件**: 系统健康巡检异常 / Agent 报告 / 用户反馈
- **执行频率**: 按需
- **预计耗时**: 视严重程度 15-120 分钟
- **负责 Agent**: COO（指挥）+ PE（修复）

## 执行步骤

### Phase 1: 检测与定性 (0-5 min)
```
🚨 故障报告
编号: INC-[YYYY]-[NNN]
时间: YYYY-MM-DD HH:MM
级别: P0/P1/P2
现象: [具体症状]
影响: [哪些功能/用户受影响]
```

### Phase 2: 止损 (5-30 min)
- **P0**: 先回滚/降级，后查因
- **P1**: 尝试快速修复，不行则降级
- 优先级: 恢复服务 > 保护数据 > 定位根因

### Phase 3: 根因分析 (15-60 min)
使用 5-Whys 方法：
```
现象: [出了什么问题]
Why 1: [为什么]
Why 2: [为什么]
Why 3: [为什么]
Why 4: [为什么]
Root Cause: [根本原因]
```

### Phase 4: 修复与预防
1. 实施修复
2. 添加监控告警（防止同类问题）
3. 更新 SOP/文档

## 输出
- 故障报告 → `memory/incidents/inc-YYYY-NNN.md`
- 修复 PR → PE
- 预防措施 → COO 跟踪



### System Health Check

# 系统健康巡检 — System Health Check

## 元信息
- **触发条件**: 每日 08:00 CST
- **执行频率**: 每天
- **预计耗时**: 8 分钟
- **负责 Agent**: COO

## 执行步骤

### Step 1: 基础设施检查 (3 min)
```
## 系统巡检 (YYYY-MM-DD)

### 服务状态
| 服务 | 状态 | CPU | 内存 | 磁盘 | 延迟 |
|------|------|-----|------|------|------|
| OpenClaw Gateway | ✅ | 12% | 45% | 23% | — |
| Freqtrade Bot 1 | ✅ | 8% | 2.1G | — | — |
| Freqtrade Bot 2 | ✅ | 6% | 1.8G | — | — |
```

### Step 2: Cron 任务检查 (3 min)
```
### Cron 任务
| 任务 | 频率 | 昨次执行 | 状态 | 耗时 |
|------|------|----------|------|------|
| 市场晨报 | 每天 08:00 | 08:00 | ✅ | 12s |
| 内容采集 | 每小时 | 14:00 | ❌ | 超时 |
```

### Step 3: 安全检查 (2 min)
- OpenClaw API key 轮换状态
- 系统异常登录/访问
- 磁盘空间预警（>80%）

## 告警规则
- 服务宕机 → 🔴 P0
- Cron 失败 > 3 次 → 🟡 P1
- CPU/内存 > 90% → 🟡 P1
- 磁盘 > 80% → 🟡 P1, > 90% → 🔴 P0

## 输出
- 巡检报告 → CEO（异常才发，正常不发）

### Crisis Escalation

# 危机升级 — Crisis Escalation

## 元信息
- **触发条件**: 以下任一条件满足时触发
- **执行频率**: 按需
- **预计耗时**: 视严重程度 5-60 分钟
- **负责 Agent**: 发现者 → CEO → 创始人

## 触发条件

| 级别 | 定义 | 示例 | 响应时间 |
|------|------|------|----------|
| 🔴 P0-Critical | 影响收入/安全/核心功能 | API key 泄露、服务宕机、资金异常 | 5 分钟内 |
| 🟠 P1-Urgent | 影响效率/用户体验 | 关键 Agent 无响应、数据管线中断 | 30 分钟内 |
| 🟡 P2-Warning | 需要关注但可容忍 | 成本超标、性能下降、外部依赖异常 | 24 小时内 |

## 执行步骤

### Step 1: 发现与定性 (0-2 分钟)
发现者（任意 Agent）执行：
1. 确认问题真实性（排除误报）
2. 判定严重等级（P0/P1/P2）
3. 按以下格式上报：

```
🚨 [P0/P1/P2] 问题简述
📍 影响范围: <哪些模块/功能>
⚡ 当前状态: <正在发生/已发生/有风险>
🛠 初步处理: <已做的>
⏰ 需要支援: <找谁>
```

### Step 2: 升级路径

```
P2 → 相关 C-Suite Agent 自行处理，CEO 抄送
P1 → 相关 C-Suite + CEO 协调，30 分钟内响应
P0 → CEO → 创始人，5 分钟内响应
    → 安全问题: 通知 CLO
    → 财务问题: 通知 CFO
    → 技术问题: 通知 PE + COO
    → 数据问题: 通知 CDO
```

### Step 3: 处理与止损
1. **P0**: 先止损，后修复。恢复服务 > 根因分析 > 预防
2. **P1**: 并行处理（一人止损，一人查因）
3. **P2**: 纳入待办，下一个工作时间窗口处理

### Step 4: 复盘与闭环
P0/P1 处理完成后：
1. 写事故报告（根因、影响、处理、预防）
2. 更新相关 SOUL.md / MEMORY.md
3. P0 必须 24h 内出复盘，P1 必须 48h 内

## 输出
- **即时**: 升级消息（群/私聊）
- **事后**: 事故报告 → `memory/` 目录
- **预防**: 流程/SOP 更新（如需要）

## 质量门禁
- [ ] P0 5 分钟内开始响应
- [ ] P0 30 分钟内恢复或降级
- [ ] 事故报告在 SLA 内产出
- [ ] 至少 1 个预防措施落地

## 异常处理
- CEO 无响应（P0）→ 直接通知创始人
- 问题范围持续扩大 → 召集跨部门紧急会议
- 无法在 SLA 内解决 → 降级方案 + 沟通预期

---

*"P0 先止血，P1 先隔离，P2 排队处理。永远不会等到'准备好'才行动。"*


### Cross Agent Handoff

# 跨 Agent 交接 — Cross-Agent Handoff

## 元信息
- **触发条件**: 任务需要从 Agent A 移交给 Agent B
- **执行频率**: 按需
- **预计耗时**: 3 分钟
- **负责 Agent**: 发起方

## 输入
- 当前任务状态（完成了什么，还差什么）
- 下一步需要做什么（具体，不是"继续做"）
- 依赖的文件/数据/凭证

## 执行步骤

### Step 1: 发起交接
Agent A 通过 `sessions_send` 发送交接消息：

```markdown
## 📋 任务交接: [任务名]

**来源**: [Agent A] → **目标**: [Agent B]
**背景**: [一句话说明为什么交接]

### ✅ 已完成
- [具体产出 1]
- [具体产出 2]

### 🎯 待完成
- [具体待做 1，含验收标准]
- [具体待做 2]

### 📁 关键文件
- `path/to/file1` — 用途说明
- `path/to/file2` — 用途说明

### ⚠️ 注意事项
- [坑/坑/坑]

### 📊 截止时间
- [日期 + 时间]
```

### Step 2: 接收确认
Agent B 收到后 10 分钟内回复：
```
✅ 已接收任务 [任务名]
📋 预计完成: [时间]
❓ 有以下疑问: [0-N 个，没有则省略]
```

### Step 3: 完成确认
Agent B 完成后通知 Agent A：
```
✅ 任务 [任务名] 已完成
📄 产出: [路径/链接]
📊 结果: [关键数据]
```

## 交接质量标准
- **绝不交空手** — 必须有具体产出（文件/代码/数据），不能只说"我做了调研"
- **绝不交模糊** — 待完成项必须有验收标准，不能说"继续优化"
- **绝不交无主** — 必须明确截止时间，不能说"有空再做"
- **绝不交黑盒** — 注意事项必须写清楚踩过的坑，不能让接手人重新踩

## 异常处理
- 接收方 10 分钟无回复 → 发起方通知 CEO
- 接收方无法独立完成 → 反向交接回发起方或请求 CEO 协调
- 任务在交接后卡住 → 最后一任负责方承担，不得反向推诿

---

*"好的交接 = 接手者不需要问任何问题就能开始干活。"*


### Daily Standup

# 每日站会 — Daily Standup

## 元信息
- **触发条件**: 每日定时 (08:00 CST) 或 CEO 手动触发
- **执行频率**: 每天
- **预计耗时**: 5 分钟/Agent
- **负责 Agent**: CEO（组织） + 各 C-Suite（汇报）
- **协作 Agent**: 全员

## 输入
- 前一日产出记录（各 agent `memory/` 目录）
- 当前进行中的任务列表
- 阻塞/依赖项

## 执行步骤

### Phase 1: CEO 收集 (08:00)
1. 遍历各 agent 的 `memory/` 最近记录
2. 检查 cron 任务执行状态（`openclaw cron list`）
3. 汇总异常/超时任务

### Phase 2: 各 Agent 汇报 (08:05)
每个 Agent 回报三件事（≤ 3 句话）：

```
📊 [角色] 昨日完成: <1件核心产出>
🎯 [角色] 今日计划: <1件核心目标>
⚠️ [角色] 阻塞/风险: <0-1个，无则省略>
```

**示例:**
```
📊 CCO 昨日完成: 3篇小红书笔记发布，1篇过500阅读
🎯 CCO 今日计划: 拆解抖音爆款5条，产出1个脚本模板
```

### Phase 3: CEO 决策 (08:15)
1. 确认优先级调整（有变更则通知相关 Agent）
2. 分配跨部门任务（指定负责人 + 截止时间）
3. 标记 P0 阻塞项（立即处理）

## 输出
- **群消息**: 站会摘要（≤ 10 行）
- **文件**: `memory/YYYY-MM-DD.md` 更新各 agent 进展

## 质量门禁
- [ ] 每个 Agent 至少报告了昨日完成和今日计划
- [ ] P0 阻塞项有明确负责人和截止时间
- [ ] 站会总时长 ≤ 15 分钟

## 异常处理
- Agent 无响应 → CEO 标记为「静默」，连续 2 天启动健康检查
- P0 冲突（多个 P0 抢资源）→ CEO 裁决优先级
- 跨部门依赖卡住 → CEO 指定协调人 + 2h SLA

---

*"站会不是汇报会，是对齐会。15 分钟解决所有信息不对称。"*


### Weekly Review

# 周度复盘 — Weekly Review

## 元信息
- **触发条件**: 每周日 20:00 CST
- **执行频率**: 每周
- **预计耗时**: 30 分钟
- **负责 Agent**: CEO
- **协作 Agent**: 全员（各 Agent 提交周报）

## 输入
- 本周各 agent `memory/` 全部记录
- 周初设定的目标列表
- 本周所有 P0/P1 任务完成情况

## 执行步骤

### Phase 1: 数据收集 (周日 20:00)
1. CEO 拉取本周所有 agent 的 git commits、任务完成记录
2. 统计核心指标：
   - 代码提交数、内容产出数、数据采集量
   - P0/P1 完成率
   - Token/成本消耗

### Phase 2: 各 Agent 周报 (周日 20:10)
每个 Agent 提交（≤ 200 字）：

```
📋 [角色] 周报
✅ 完成: (数字) 件核心产出
❌ 未完成: (哪些 + 原因)
📈 数据: (关键数字)
💡 下周重点: (1-3 件)
🔧 改进: (本周学到的)
```

### Phase 3: CEO 综合评估 (周日 20:25)
1. **目标达成率** — 本周 OKR 完成百分比
2. **效率分析** — 哪些 Agent 高效/低效，原因分析
3. **资源瓶颈** — 是否需要新增 skill / 调整分工
4. **风险预警** — 下周可能出问题的事项

### Phase 4: 下周规划 (周日 20:30)
1. CEO 设定下周 P0/P1
2. 分配跨部门项目
3. 更新团队看板

## 输出
- **群消息**: 周报摘要 + 下周重点（≤ 20 行）
- **文件**: `memory/YYYY-MM-DD.md` 周复盘记录

## 质量门禁
- [ ] 所有 Agent 提交周报
- [ ] P0 完成率 ≥ 80%
- [ ] 下周 P0 明确且 ≤ 3 个
- [ ] 有至少 1 个改进措施落地

## 异常处理
- Agent 周报缺失 → 记录为「周度沉默」，连续 2 周升级为 P1
- P0 完成率 < 50% → CEO 重新评估可行性，调整目标或增派资源
- 跨部门项目连续 2 周无进展 → 取消或降级

---

*"不复盘的工作等于没做。数据说话，数字不撒谎。"*
