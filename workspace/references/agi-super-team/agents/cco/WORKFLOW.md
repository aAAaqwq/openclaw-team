# WORKFLOW.md — Ives (CCO) 标准工作流程

## 内容创作流程
1. **选题** → 从热点池/用户反馈/数据趋势中选择方向
2. **减法设计** → 确定核心信息，删除一切非必要元素
3. **3秒测试** → 标题/封面是否能在3秒内抓住注意力？
4. **平台适配** → 同一核心内容适配不同平台形态
5. **发布** → 选择最佳时间窗口
6. **数据回收** → 追踪关键指标（点击率、完播率、互动率）
7. **迭代** → 基于数据优化下一轮内容

## 协作触发
- 需要深度素材 → Feynman(CRO)
- 需要投放配合 → Ogilvy(CMO)
- 需要效果数据 → Silver(CDO)

### Content Calendar

# 内容日历规划 — Content Calendar

## 元信息
- **触发条件**: 每周一固定执行
- **执行频率**: 每周
- **预计耗时**: 20 分钟
- **负责 Agent**: CCO
- **协作 Agent**: CMO（SEO 趋势）、CQO（相关热点）

## 输入
- 上周内容数据回顾
- 本周热点/节日/行业事件日历
- 平台流量趋势（搜索词、热门话题）
- 产品/项目发布计划

## 执行步骤

### Step 1: 回顾上周 (3 min)
```
| 平台 | 发布数 | 平均数据 | 最佳表现 | 最差表现 |
|------|--------|----------|----------|----------|
```

### Step 2: 选题规划 (10 min)

**选题来源（优先级排序）:**
1. **数据驱动**: 上周数据好的话题 → 延伸/深挖
2. **热点借势**: 本周热点/节日/行业大事件
3. **SEO 关键词**: 搜索量上升的词 → 内容承接
4. **系列延续**: 未完结的系列内容
5. **产品配合**: 新功能/新项目 → 对应内容

**分配规则:**
- 小红书: 5-7 篇/周（图文为主，短平快）
- 抖音: 3-5 篇/周（视频，30-60s）
- 微信: 1-2 篇/周（深度长文）
- 知乎: 1-2 篇/周（专业回答/文章）
- X: 3-5 条/周（观点/ threads）

### Step 3: 排期输出 (7 min)
输出本周排期表：

```
## 本周内容日历 (MM-DD ~ MM-DD)

### 周一
- [08:30] 小红书: [选题] [标签]
- [12:00] 抖音: [选题] [时长]

### 周二
...

### 重点关注
- [节日/热点]: [应对策略]
- [系列连载]: [第N篇]
- [测试内容]: [AB测试项]
```

## 输出
- **日历文件**: `output/content-calendar/week-YYYY-WW.md`
- **选题池**: 更新 `output/content-queue.md`
- **同步**: 通知团队本周内容重点

## 质量门禁
- [ ] 每天至少 1 篇跨平台内容
- [ ] 至少 1 个热点借势内容
- [ ] 至少 1 个系列连载续更
- [ ] 有 AB 测试项（至少 1 个新格式/新话题尝试）

---

*"好的内容日历 = 70% 常规 + 20% 热点 + 10% 实验。稳定产出是基础，爆款是概率。"*


### Content Pipeline

# 内容发布管线 — Content Pipeline

## 元信息
- **触发条件**: 每日内容生产计划或 CEO 派发任务
- **执行频率**: 每天
- **预计耗时**: 单篇 15-30 分钟（批量 2-3 小时）
- **负责 Agent**: CCO
- **协作 Agent**: CDO（数据支撑）、PE（发布工具）、CMO（SEO 优化）

## 输入
- 内容选题（来自内容日历或热点追踪）
- 平台要求（字数、格式、标签规范）
- 素材（数据、截图、引用）

## 执行步骤

### Phase 1: 选题确认 (5 min)
1. 检查内容日历 → 确认今日选题
2. 检查各平台热点 → 有突发热点则优先调整
3. 确定平台优先级：小红书 > 抖音 > 微信 > 知乎 > X

### Phase 2: 内容生产 (15-20 min/篇)
每个平台按规范执行：

**小红书图文:**
```
选题 → 标题(3选1) → 正文(300-800字) → 封面图 → 标签(5-8个) → 发布
```

**抖音视频:**
```
选题 → 脚本(30-60s) → 素材准备 → AI生成/剪辑 → 封面 → 标题+话题 → 发布
```

**微信公众号:**
```
选题 → 大纲 → 正文(1500-3000字) → 排版 → 封面 → 摘要 → 定时发布
```

### Phase 3: 分发与适配 (10 min)
1. 一稿多发 → 不同平台格式适配
   - 小红书: 短句 + emoji + 封面
   - 抖音: 口播脚本 + BGM
   - 知乎: 长文 + 引用 + 数据
2. SEO 优化 → 关键词植入 + 标题优化

### Phase 4: 发布后监控 (5 min)
1. 发布后 1h 检查初始数据（阅读/播放/互动）
2. 数据异常（低于平时 50%）→ 标记排查
3. 爆款信号（超平时 200%）→ 通知 CMO 加投

## 输出
- **已发布内容**: 各平台链接/截图
- **数据报告**: `output/daily-content-stats.md`
- **明日选题**: `output/content-queue.md`

## 质量门禁
- [ ] 每篇内容发布前检查格式合规（标题/字数/标签）
- [ ] 封面图与内容匹配
- [ ] 每日至少产出 3 篇跨平台内容
- [ ] 发布后 1h 有数据回看

## 批量优化
- 主题系列化: 同一选题 → 拆成 5-7 篇系列，一周分发
- 模板化: 常用结构（盘点类/教程类/观点类）→ SKILL 自动填充
- 时间窗口: 早 8-9（通勤）、午 12-13（午休）、晚 20-22（睡前）

## 异常处理
- 平台限流 → 切换平台或暂停 30 分钟
- 封面图生成失败 → 使用备用模板或纯色背景
- 内容审核不通过 → 检查敏感词，修改后重发

---

*"内容生产是流水线，不是艺术创作。标准化 → 批量化 → 数据优化。"*


### Viral Debrief

# 爆款复盘 — Viral Content Debrief

## 元信息
- **触发条件**: 单篇内容数据超过平时 3 倍，或周度定期执行
- **执行频率**: 每周 + 实时触发
- **预计耗时**: 15 分钟/篇
- **负责 Agent**: CCO
- **协作 Agent**: CMO（增长归因）、CDO（数据采集）

## 输入
- 爆款内容原文 + 平台数据截图
- 同时间段其他内容数据（对照组）
- 平台推荐算法特征分析

## 执行步骤

### Step 1: 数据采集 (3 min)
1. 导出爆款内容全生命周期数据:
   - 发布时间、初始数据、增长曲线、峰值时间、衰减曲线
2. 导出同期对照组数据（发布前后各 5 篇）

### Step 2: 拆解分析 (8 min)
逐维度分析**为什么爆了**：

| 维度 | 分析项 | 结论 |
|------|--------|------|
| 标题 | 关键词/情绪/数字/反差 | 哪个元素吸了睛 |
| 封面 | 配色/人物/文字/构图 | 点击率驱动因素 |
| 内容结构 | 开头钩子/中间节奏/结尾CTA | 完播/完读率为什么高 |
| 时机 | 发布时间/平台流量/热点借势 | 是否蹭到了流量 |
| 互动 | 评论引导/争议点/共鸣 | 互动率异常原因 |

### Step 3: 提炼公式 (4 min)
从拆解中提取**可复制的公式**：

```
## 复制公式: [类型]

✅ 可复制要素:
1. [具体技巧 1]
2. [具体技巧 2]
3. [具体技巧 3]

⚠️ 不可复制要素:
1. [时机/运气/账号权重等]

📝 模板化:
[生成一个可直接套用的模板]
```

## 输出
- **爆款档案**: `output/viral/[date]-[platform]-[topic].md`
- **复制模板**: 可直接套用的标题/封面/结构模板
- **更新**: 写入 SKILL.md 或 `memory/` 供后续复用

## 质量门禁
- [ ] 数据对比有对照组（不是孤立看一个数字）
- [ ] 至少提炼 2 个可复制要素
- [ ] 生成了 1 个可套用模板
- [ ] 更新了内容策略文档

---

*"爆款不是运气，是模式。拆得够细，就能复制。拆一次不够，拆十次就有公式。"*

## 🔄 Shared Workflows

### Crisis Escalation

# 危机升级 — Crisis Escalation

## 元信息
- **触发条件**: 以下任一条件满足时触发
- **执行频率**: 按需
- **预计耗时**: 视严重程度 5-60 分钟
- **负责 Agent**: 发现者 → CEO → 创始人

## 触发条件

| 级别 | 定义 | 示例 | 响应时间 |
|------|------|------|----------|
| 🔴 P0-Critical | 影响收入/安全/核心功能 | API key 泄露、服务宕机、资金异常 | 5 分钟内 |
| 🟠 P1-Urgent | 影响效率/用户体验 | 关键 Agent 无响应、数据管线中断 | 30 分钟内 |
| 🟡 P2-Warning | 需要关注但可容忍 | 成本超标、性能下降、外部依赖异常 | 24 小时内 |

## 执行步骤

### Step 1: 发现与定性 (0-2 分钟)
发现者（任意 Agent）执行：
1. 确认问题真实性（排除误报）
2. 判定严重等级（P0/P1/P2）
3. 按以下格式上报：

```
🚨 [P0/P1/P2] 问题简述
📍 影响范围: <哪些模块/功能>
⚡ 当前状态: <正在发生/已发生/有风险>
🛠 初步处理: <已做的>
⏰ 需要支援: <找谁>
```

### Step 2: 升级路径

```
P2 → 相关 C-Suite Agent 自行处理，CEO 抄送
P1 → 相关 C-Suite + CEO 协调，30 分钟内响应
P0 → CEO → 创始人，5 分钟内响应
    → 安全问题: 通知 CLO
    → 财务问题: 通知 CFO
    → 技术问题: 通知 PE + COO
    → 数据问题: 通知 CDO
```

### Step 3: 处理与止损
1. **P0**: 先止损，后修复。恢复服务 > 根因分析 > 预防
2. **P1**: 并行处理（一人止损，一人查因）
3. **P2**: 纳入待办，下一个工作时间窗口处理

### Step 4: 复盘与闭环
P0/P1 处理完成后：
1. 写事故报告（根因、影响、处理、预防）
2. 更新相关 SOUL.md / MEMORY.md
3. P0 必须 24h 内出复盘，P1 必须 48h 内

## 输出
- **即时**: 升级消息（群/私聊）
- **事后**: 事故报告 → `memory/` 目录
- **预防**: 流程/SOP 更新（如需要）

## 质量门禁
- [ ] P0 5 分钟内开始响应
- [ ] P0 30 分钟内恢复或降级
- [ ] 事故报告在 SLA 内产出
- [ ] 至少 1 个预防措施落地

## 异常处理
- CEO 无响应（P0）→ 直接通知创始人
- 问题范围持续扩大 → 召集跨部门紧急会议
- 无法在 SLA 内解决 → 降级方案 + 沟通预期

---

*"P0 先止血，P1 先隔离，P2 排队处理。永远不会等到'准备好'才行动。"*


### Cross Agent Handoff

# 跨 Agent 交接 — Cross-Agent Handoff

## 元信息
- **触发条件**: 任务需要从 Agent A 移交给 Agent B
- **执行频率**: 按需
- **预计耗时**: 3 分钟
- **负责 Agent**: 发起方

## 输入
- 当前任务状态（完成了什么，还差什么）
- 下一步需要做什么（具体，不是"继续做"）
- 依赖的文件/数据/凭证

## 执行步骤

### Step 1: 发起交接
Agent A 通过 `sessions_send` 发送交接消息：

```markdown
## 📋 任务交接: [任务名]

**来源**: [Agent A] → **目标**: [Agent B]
**背景**: [一句话说明为什么交接]

### ✅ 已完成
- [具体产出 1]
- [具体产出 2]

### 🎯 待完成
- [具体待做 1，含验收标准]
- [具体待做 2]

### 📁 关键文件
- `path/to/file1` — 用途说明
- `path/to/file2` — 用途说明

### ⚠️ 注意事项
- [坑/坑/坑]

### 📊 截止时间
- [日期 + 时间]
```

### Step 2: 接收确认
Agent B 收到后 10 分钟内回复：
```
✅ 已接收任务 [任务名]
📋 预计完成: [时间]
❓ 有以下疑问: [0-N 个，没有则省略]
```

### Step 3: 完成确认
Agent B 完成后通知 Agent A：
```
✅ 任务 [任务名] 已完成
📄 产出: [路径/链接]
📊 结果: [关键数据]
```

## 交接质量标准
- **绝不交空手** — 必须有具体产出（文件/代码/数据），不能只说"我做了调研"
- **绝不交模糊** — 待完成项必须有验收标准，不能说"继续优化"
- **绝不交无主** — 必须明确截止时间，不能说"有空再做"
- **绝不交黑盒** — 注意事项必须写清楚踩过的坑，不能让接手人重新踩

## 异常处理
- 接收方 10 分钟无回复 → 发起方通知 CEO
- 接收方无法独立完成 → 反向交接回发起方或请求 CEO 协调
- 任务在交接后卡住 → 最后一任负责方承担，不得反向推诿

---

*"好的交接 = 接手者不需要问任何问题就能开始干活。"*


### Daily Standup

# 每日站会 — Daily Standup

## 元信息
- **触发条件**: 每日定时 (08:00 CST) 或 CEO 手动触发
- **执行频率**: 每天
- **预计耗时**: 5 分钟/Agent
- **负责 Agent**: CEO（组织） + 各 C-Suite（汇报）
- **协作 Agent**: 全员

## 输入
- 前一日产出记录（各 agent `memory/` 目录）
- 当前进行中的任务列表
- 阻塞/依赖项

## 执行步骤

### Phase 1: CEO 收集 (08:00)
1. 遍历各 agent 的 `memory/` 最近记录
2. 检查 cron 任务执行状态（`openclaw cron list`）
3. 汇总异常/超时任务

### Phase 2: 各 Agent 汇报 (08:05)
每个 Agent 回报三件事（≤ 3 句话）：

```
📊 [角色] 昨日完成: <1件核心产出>
🎯 [角色] 今日计划: <1件核心目标>
⚠️ [角色] 阻塞/风险: <0-1个，无则省略>
```

**示例:**
```
📊 CCO 昨日完成: 3篇小红书笔记发布，1篇过500阅读
🎯 CCO 今日计划: 拆解抖音爆款5条，产出1个脚本模板
```

### Phase 3: CEO 决策 (08:15)
1. 确认优先级调整（有变更则通知相关 Agent）
2. 分配跨部门任务（指定负责人 + 截止时间）
3. 标记 P0 阻塞项（立即处理）

## 输出
- **群消息**: 站会摘要（≤ 10 行）
- **文件**: `memory/YYYY-MM-DD.md` 更新各 agent 进展

## 质量门禁
- [ ] 每个 Agent 至少报告了昨日完成和今日计划
- [ ] P0 阻塞项有明确负责人和截止时间
- [ ] 站会总时长 ≤ 15 分钟

## 异常处理
- Agent 无响应 → CEO 标记为「静默」，连续 2 天启动健康检查
- P0 冲突（多个 P0 抢资源）→ CEO 裁决优先级
- 跨部门依赖卡住 → CEO 指定协调人 + 2h SLA

---

*"站会不是汇报会，是对齐会。15 分钟解决所有信息不对称。"*


### Weekly Review

# 周度复盘 — Weekly Review

## 元信息
- **触发条件**: 每周日 20:00 CST
- **执行频率**: 每周
- **预计耗时**: 30 分钟
- **负责 Agent**: CEO
- **协作 Agent**: 全员（各 Agent 提交周报）

## 输入
- 本周各 agent `memory/` 全部记录
- 周初设定的目标列表
- 本周所有 P0/P1 任务完成情况

## 执行步骤

### Phase 1: 数据收集 (周日 20:00)
1. CEO 拉取本周所有 agent 的 git commits、任务完成记录
2. 统计核心指标：
   - 代码提交数、内容产出数、数据采集量
   - P0/P1 完成率
   - Token/成本消耗

### Phase 2: 各 Agent 周报 (周日 20:10)
每个 Agent 提交（≤ 200 字）：

```
📋 [角色] 周报
✅ 完成: (数字) 件核心产出
❌ 未完成: (哪些 + 原因)
📈 数据: (关键数字)
💡 下周重点: (1-3 件)
🔧 改进: (本周学到的)
```

### Phase 3: CEO 综合评估 (周日 20:25)
1. **目标达成率** — 本周 OKR 完成百分比
2. **效率分析** — 哪些 Agent 高效/低效，原因分析
3. **资源瓶颈** — 是否需要新增 skill / 调整分工
4. **风险预警** — 下周可能出问题的事项

### Phase 4: 下周规划 (周日 20:30)
1. CEO 设定下周 P0/P1
2. 分配跨部门项目
3. 更新团队看板

## 输出
- **群消息**: 周报摘要 + 下周重点（≤ 20 行）
- **文件**: `memory/YYYY-MM-DD.md` 周复盘记录

## 质量门禁
- [ ] 所有 Agent 提交周报
- [ ] P0 完成率 ≥ 80%
- [ ] 下周 P0 明确且 ≤ 3 个
- [ ] 有至少 1 个改进措施落地

## 异常处理
- Agent 周报缺失 → 记录为「周度沉默」，连续 2 周升级为 P1
- P0 完成率 < 50% → CEO 重新评估可行性，调整目标或增派资源
- 跨部门项目连续 2 周无进展 → 取消或降级

---

*"不复盘的工作等于没做。数据说话，数字不撒谎。"*
