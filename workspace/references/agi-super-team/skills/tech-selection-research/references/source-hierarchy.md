# Source Hierarchy

Use current, high-signal sources. Prefer primary material when deciding whether a technology is suitable for adoption.

## Priority Order

### Tier 1: Primary Sources

- Official docs
- Official release notes and changelogs
- Official pricing and licensing pages
- Roadmaps, RFCs, proposals
- Maintainer talks, official blog posts, foundation docs

Use these first for:

- Supported features
- Deployment models
- Version status
- Licensing and packaging
- Roadmap direction

### Tier 2: High-Quality Secondary Sources

- Thoughtworks Radar
- InfoQ
- CNCF or other foundation material
- Major engineering blogs from companies running the technology in production

Use these for:

- Adoption patterns
- Failure modes
- Organizational tradeoffs
- Market positioning and maturity signals

### Tier 3: Supplemental Sources

- Tutorials
- Community comparisons
- Forum discussions

Use only to clarify implementation details or common operator concerns.

## Required Source Handling

- Separate current facts from older background
- Quote sparingly; summarize the substance
- Mark uncertain claims as `Needs PoC`
- If pricing or release state is relevant, verify with current official pages

## Common Failure Modes

- Popularity bias: over-recommending fashionable tools
- Freshness bias: assuming the newest feature changes the adoption decision
- Vendor bias: repeating homepage claims without constraints
- Benchmark fantasy: citing performance conclusions without reproducible context
