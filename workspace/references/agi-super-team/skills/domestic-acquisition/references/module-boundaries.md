# 模块边界规范

用于 `domestic-acquisition` 包内各子模块统一声明职责边界，避免重复建设、职责重叠与验收口径不一致。

## 边界原则

- 单模块只解决一个核心问题，不同时承担“采集 + 转化 + 自动触达”多段职责。
- 输入输出必须可交接：上游产出应能被下游直接消费，不依赖口头补充。
- 模块描述必须同时写清：边界内、边界外、上游输入模块、下游输出模块。
- 通用规范抽到 `references/`；单模块只写自身特有逻辑。

## 当前模块职责划分

| 模块 | 负责 | 不负责 |
|---|---|---|
| `lead-mining` | 企业线索采集、字段标准化、来源留痕、初步去重 | 深度画像分析、转化话术、自动触达 |
| `customer-profile` | 在线索基础上补充画像字段、客户分层、优先级判断 | 原始大规模采集、自动外呼/群发 |
| `social-funnel` | 公域内容引流、渠道承接、入口设计 | 企业名单采集、私聊成交 |
| `private-traffic` | 私域承接、跟进节奏、话术沉淀 | 大规模外部线索采集 |
| `competitor-intel` | 竞品获客动作、渠道与内容样本监控 | 直接客户触达执行 |
| `outreach-automation` | 基于已结构化线索执行触达准备与自动化编排 | 原始线索发现、深度内容运营 |

## 验收检查清单

每个子模块验收时至少检查：

1. 是否只解决一个核心问题。
2. 是否明确写出边界内 / 边界外。
3. 是否声明上游输入模块与下游输出模块。
4. 输出是否能被下一模块直接消费。
5. 是否把跨模块规范沉淀到 `references/`，而不是写死在单模块。
